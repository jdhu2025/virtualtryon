# AI衣厨 · MVP 产品规格文档 v3.0

> 文档版本：v3.0
> 更新日期：2026-04-16
> 文档状态：MVP 方向重构版
> 适用范围：全球市场，不预设单一区域

---

## 一、产品判断

### 1.1 这次要解决的不是“搭配很多”，而是“今天别再纠结”

用户真正的高频痛点不是没有衣服，也不是没有灵感，而是：

- 早上要出门时，不想再花脑力决定穿什么
- 明明衣服不少，但总觉得“没有能直接穿出去的一套”
- 想显瘦、想省事、想舒服、想体面，但很难快速把这些要求转成搭配

### 1.2 产品一句话

**帮用户用已有衣服，在几分钟内得到一套今天能直接穿出去的方案，并在需要时用本人照片确认这套大概穿上什么样。**

### 1.3 MVP 只保留两个核心功能

1. **快速入衣橱**
   先让用户用极低成本录入少量常穿单品，不要求一开始整理完整个衣柜。
2. **今天穿什么**
   基于用户刚录入的少量衣物和一句自然语言需求，给出一套可直接执行的穿搭建议，并允许轻量调整与按需试穿确认。

## 二、设计原则

### 2.1 先给价值，再要求整理

- 不要求用户先录完整个衣柜
- 先录入常穿的 15-30 件，甚至先录 3-5 件也要能开始
- 录完一件就能继续，录完几件就能立刻试搭

### 2.2 少给选择，减少决策疲劳

- 默认只给 **1 套主方案**
- 用户主动点击时，再给 **最多 2 套替代方案**
- 不出现“20 套图墙”式输出

### 2.3 用真实语言，不用平台语言

- 用户输入应支持：
  - `想显瘦一点`
  - `今天不想费劲`
  - `要舒服但别太随便`
  - `办公室空调冷`
  - `给我一套不用想的`
- 不强迫用户先选择“通勤 / 约会 / 聚会”等平台分类

### 2.4 允许不精确

- 用户不必填写完整颜色、季节、风格标签
- AI 能识别就自动带出，识别不准也不阻塞保存
- 除图片外，MVP 只强制一个最小确认字段：`类别`

### 2.5 反馈必须足够轻

- 推荐结果上的反馈只保留：
  - `喜欢`
  - `不喜欢`
  - `今天不适合`
  - `换件上衣`
  - `再来一套`
- 不设计长问卷式反馈

### 2.6 默认尊重人的惰性

- 支持重复穿
- 支持“复制昨天逻辑”
- 支持“今天别复杂，给我一套最省事的”
- 不把“频繁换新”当成产品目标

## 三、MVP 范围

### 3.1 Core Function A：快速入衣橱

目标：让用户在 **3 分钟内完成首批录入**，并尽快进入推荐环节。

#### 必须实现

- 支持拍照或上传单品图片
- AI 自动识别：
  - 类别
  - 颜色
  - 一句简短描述
- 用户只需确认或修正 `类别` 即可保存
- 颜色、风格、季节、场景标签默认折叠为“补充信息”，非必填
- 单品保存后立即可用于推荐

#### 交互要求

- 首页主流程中应出现明显入口：`先加几件常穿的`
- 当衣柜为空时，不显示复杂管理面板，优先引导用户先录 3 件
- 每次保存成功后，优先给两个动作：
  - `继续加一件`
  - `先试着搭一套`

#### 最小成功标准

- 用户录完 3 件单品后即可进入“今天穿什么”
- 即使标签不完整，也不阻塞推荐

### 3.2 Core Function B：今天穿什么

目标：在用户已有少量衣物时，快速给出 **1 套可信、可穿、能解释清楚** 的方案，并在用户确认方向后，再对这 1 套进行本人照片试穿确认。

#### 必须实现

- 支持一句自然语言输入
- 支持少量快捷短句，但文案必须接近真实表达
- 默认输出：
  - 1 套主方案
  - 1 句理由
  - 涉及的单品列表
  - 1 个试穿确认动作：`试穿这套`
  - 1 个轻量操作（如 `换件上衣`）
- 用户主动点击后，才继续生成替代方案，最多再给 2 套

#### 输入示例

- `今天想显瘦一点`
- `不想穿得太正式，但要见人`
- `今天不想费劲`
- `给我一套舒服但别太随便的`
- `复制昨天那种省事逻辑`

#### 输出要求

- 输出像“今天能穿出去的一套”，不是灵感拼贴
- 理由必须短而具体，例如：
  - `上深下直更利落，整体显瘦，也不会太用力`
  - `这套舒服但不松垮，适合今天想省脑力的状态`
- 用户默认先看到“方案卡”，不是先等待全量试衣图生成
- 推荐优先级：
  - 可穿性
  - 符合需求
  - 降低重复决策
  - 再考虑风格多样性

#### 交互形态

`今天穿什么` 页不是纯聊天页，也不是多套图流，而是一个 **单任务决策器**：

1. 用户输入一句真实需求
2. 系统先返回 1 套主方案卡
3. 用户如果觉得方向对，再点击 `试穿这套`
4. 系统只对这 1 套生成本人照片预览
5. 用户再做轻反馈或局部替换

#### 试穿确认要求

- `试穿这套` 属于第二核心功能内的确认动作，不单独拆成第三核心功能
- 只有在用户主动点击后才生成试穿结果
- 默认只对当前主方案生成，不对多套同时生成
- 若试穿失败或质量不稳定，仍保留原方案卡和理由，不让主流程中断
- MVP 可接受“有限保真、有限品类、固定姿态”的试穿确认，不要求全场景高保真

#### 轻反馈要求

- 每套推荐结果下固定提供：
  - `喜欢`
  - `不喜欢`
  - `今天不适合`
  - `换件上衣`
  - `再来一套`

#### 默认容忍不完整衣柜

- 当衣柜只有 3-5 件时，也要能工作
- 若单品不足以形成完整方案，应返回：
  - 当前能搭出的最接近方案
  - 还缺哪一类单品
  - 一个下一步动作，例如 `再加一条裤子会更准`

## 四、MVP 明确不做

以下能力不进入 MVP 成功标准：

- ❌ 对所有候选方案默认进行高保真虚拟试衣
- ❌ 全品类、全姿态、全场景一致性的高保真虚拟试衣
- ❌ 3 张效果图并排对比
- ❌ 多张人像选择与多图生成
- ❌ 社交分享卡片与品牌水印
- ❌ 连续打卡、徽章、游戏化
- ❌ 内容社区
- ❌ 电商导购与购买跳转
- ❌ 复杂标签管理后台
- ❌ 长对话式聊天历史

说明：
MVP 的目标是先证明“少量录入 + 今日建议 + 单套按需试穿确认”能否形成稳定使用，而不是证明复杂图像生成能力。

## 五、最小用户旅程

### 5.1 首次使用

```
打开首页
  ↓
看到两个主动作：今天穿什么 / 先加几件常穿的
  ↓
若衣柜为空，优先引导先录 3 件
  ↓
拍照上传 3-5 件单品
  ↓
AI 自动识别，用户只确认类别即可保存
  ↓
立即进入“今天穿什么”
```

### 5.2 每日使用

```
打开首页
  ↓
点击“今天穿什么”
  ↓
输入一句真实需求
  ↓
得到 1 套主方案 + 1 句理由
  ↓
若方向正确，点击“试穿这套”
  ↓
系统生成本人照片预览
  ↓
用户轻反馈：喜欢 / 不喜欢 / 今天不适合 / 换件 / 再来一套
  ↓
若满意，可标记为今天穿了
```

### 5.3 惰性友好路径

```
昨天穿得不错
  ↓
今天只想少想一点
  ↓
点击“复制昨天逻辑”
  ↓
系统沿用相似搭配结构，避开完全重复或按需允许重复
```

## 六、功能规格

### 6.1 首页信息架构

首页只服务于两个核心动作，不应再平均分配注意力给太多模块。

#### 首屏必须有

- 主按钮：`今天穿什么`
- 次按钮：`先加几件常穿的`
- 一条低压力文案：
  - `不用先整理完整个衣柜，先从常穿的几件开始`

#### 首页可选信息

- 当前已录入件数
- 最近一次喜欢的搭配
- 一个快捷入口：`复制昨天逻辑`

#### 首页不应优先展示

- 大段功能介绍
- 多模块宫格导航
- 分享入口
- 复杂统计图

### 6.2 快速入衣橱

#### 输入方式

- 手机拍照
- 相册上传

#### 必填 / 选填

| 字段 | 是否必填 | 说明 |
|------|---------|------|
| 图片 | 必填 | 必须有单品图 |
| 类别 | 必填 | AI 预填，用户只需确认 |
| 颜色 | 选填 | AI 可预填 |
| 风格标签 | 选填 | 默认折叠 |
| 季节 | 选填 | 默认折叠 |
| 描述 | 选填 | AI 生成一句描述，用户可不改 |

#### 成功标准

- AI 识别失败时，用户仍可手动选择类别后保存
- 单品保存完成后，无需再补完标签即可参与推荐

#### 文案原则

- 使用 `先加一件常穿的`
- 不使用 `完善你的数字衣橱档案`

### 6.3 今天穿什么

#### 输入区设计

- 一个自然语言输入框
- 3-5 个贴近日常表达的快捷短句

推荐示例：

- `想显瘦一点`
- `今天不想费劲`
- `要舒服但别太随便`
- `给我一套不用想的`
- `复制昨天逻辑`

不推荐示例：

- `请选择场景`
- `请选择风格`
- `请选择色彩倾向`

#### 输出区设计

默认只显示 1 套主方案，包含：

- 方案标题
- 一句话理由
- 单品清单
- `试穿这套` 按钮
- 轻量反馈按钮
- 一个局部替换操作

用户点击 `再来一套` 后，才显示下一套。

#### 结果卡之后的确认层

点击 `试穿这套` 后，展开确认层，包含：

- 当前方案对应的本人照片试穿结果
- `喜欢`
- `换件上衣`
- `今天不适合`
- `再来一套`

确认层的目标是帮助用户“最后拍板”，不是重新进入复杂浏览。

#### 推荐逻辑优先级

1. 满足用户当下表达的需求
2. 尽量用已有高频、低风险单品
3. 尽量减少用户再做二次决策
4. 再考虑风格变化

### 6.4 轻量反馈

反馈不是独立主功能，而是第二核心功能的内嵌动作。

#### 反馈按钮

- `喜欢`
- `不喜欢`
- `今天不适合`
- `换件上衣`
- `再来一套`

#### MVP 反馈用途

- `喜欢`：提高相似逻辑权重
- `不喜欢`：降低相似逻辑权重
- `今天不适合`：视为临时上下文，不永久惩罚
- `换件上衣`：局部重算，避免整套推翻
- `再来一套`：生成下一方案，但最多 2 次

### 6.5 最近穿过 / 复制昨天逻辑

这是“今天穿什么”的支持能力，不单独列为核心功能。

#### MVP 要求

- 允许用户标记“今天穿了这套”
- 系统能记住最近穿过的单品或搭配结构
- 支持 `复制昨天逻辑`
- 支持“避免完全重复”，但不可强迫避重

## 七、哪些功能比较难实现

### 7.1 超高难度：不建议进入 MVP

| 功能 | 难度 | 为什么难 | MVP 建议 |
|------|------|---------|---------|
| 对所有候选方案做高保真虚拟试衣 | 极高 | 受人像姿态、衣物品类、遮挡、材质、叠穿影响极大，结果很难稳定可信，且会拖慢主流程 | 不进入 MVP |
| 多品类试穿效果一致性 | 极高 | 鞋、包、帽子、外套、裙装对位置与比例要求完全不同 | 直接移出 MVP |
| “像真人造型师一样一次就懂你” 的长期个性化模型 | 极高 | 需要长期偏好、体感、场景、重复率、体型状态、心理预期等多维记忆 | MVP 先用轻反馈 + 简单记忆 |

### 7.2 高难度：可以做，但不要作为首版成败标准

| 功能 | 难度 | 为什么难 | 建议 |
|------|------|---------|------|
| 任意图片自动精准入衣橱 | 高 | 平铺、挂拍、上身图、商品图、截图质量差异大，AI 识别稳定性有限 | 先做到“可保存、可修正”，别追求全自动精确 |
| 一次就准的穿搭推荐 | 高 | 用户表达常含模糊约束，且真实决策会受天气、体感、洗护状态等影响 | 先保证主方案可信，再提供轻量替换 |
| 自动避免重复又不让用户烦 | 高 | 用户并不总想创新，有时反而想重复安全方案 | 先做“允许重复 + 可复制昨天逻辑” |
| 单套按需试穿确认 | 高 | 即使只试 1 套，也会受姿态、遮挡、图像质量影响 | 先限制品类、姿态和触发时机 |

### 7.3 中等难度：适合尽快做

| 功能 | 难度 | 为什么值得先做 | 建议 |
|------|------|---------------|------|
| 一拍即走式反馈 | 中 | 数据成本低，能快速改善推荐 | 放进 MVP |
| 局部替换（换件上衣/换双鞋） | 中 | 比整套重算更贴近真实决策 | 放进 MVP |
| 最近穿过记忆 | 中 | 能显著提升“像懂我”的感觉 | MVP 可做轻量版 |

## 八、当前项目与新 SPEC 的关系

### 8.1 当前已有能力，可直接复用

- `/api/analyze-cloth`
  已有 AI 识别能力，可继续作为“快速入衣橱”的基础
- `/wardrobe/add`
  已有上传、AI 预填、手动修正能力，可改为更轻的表单
- `/api/wardrobe` 与本地存储
  已具备最小衣柜数据闭环

### 8.2 当前已有能力，但不再是 MVP 主线

- `/api/generate-outfit-image`
  当前偏“图像生成 + 虚拟试衣”，建议改造为“主方案确认后的单套试穿能力”
- `chat/page.tsx`
  当前是“对话 + 多套结果”模式，建议改为“今天穿什么”单任务页面
- 多人像选择
  现阶段不应继续扩展

### 8.3 当前缺失但 MVP 应补上

- 首页信息架构重做，聚焦两个主动作
- “衣柜为空但可快速开始”的低压力引导
- 推荐结果上的轻反馈按钮
- 主方案卡上的 `试穿这套` 确认动作
- “复制昨天逻辑”与“今天穿了这套”的轻量记忆
- 推荐结果默认只出 1 套，而不是一开始就铺开多套

## 九、推荐的数据与状态变更

### 9.1 单品最小数据

MVP 单品推荐至少依赖以下字段：

| 字段 | 用途 |
|------|------|
| id | 唯一标识 |
| user_id | 数据隔离 |
| image_url | 展示 |
| category | 参与搭配的最小结构字段 |
| color | 辅助搭配，可选 |
| ai_description | 推荐理由文案辅助 |
| created_at | 排序 |

### 9.2 推荐新增字段

为了支持“今天穿什么”与轻记忆，建议新增：

| 字段 | 用途 |
|------|------|
| last_worn_at | 最近穿着时间 |
| wear_count | 穿着次数 |
| review_status | 是否仅快速保存、待后续补充 |

### 9.3 反馈最小记录

建议为推荐结果增加最小反馈记录：

| 字段 | 用途 |
|------|------|
| outfit_id | 关联推荐结果 |
| feedback_type | `like` / `dislike` / `not_today` / `wore_today` |
| created_at | 时间 |

## 十、MVP 成功指标

### 10.1 用户价值指标

- 首次打开后，**3 分钟内** 能得到第一套推荐
- 首次推荐前，用户最多只需录入 **3-5 件** 单品
- 每次“今天穿什么”的默认结果不超过 **1 套主方案**
- 用户能在 **1 次点击** 内表达喜欢或不喜欢

### 10.2 产品判断指标

- 有多少用户在录入少量单品后，第二天还会回来再问一次“今天穿什么”
- 有多少用户会主动使用 `复制昨天逻辑`
- 有多少用户会用 `换件上衣` 而不是直接退出

如果这些指标不成立，说明用户并没有把产品当成“省脑力工具”。

## 十一、阶段规划

### Phase 1：证明最小闭环

- 快速入衣橱
- 今天穿什么
- 单套按需试穿确认
- 轻反馈
- 最近穿过记忆

### Phase 2：提升可信度

- 更好的局部替换
- 更稳定的单套试穿质量
- 更像用户说话方式的输入建议
- 基于反馈的简单偏好学习

### Phase 3：谨慎评估是否重启图像能力

只有当以下条件成立时，才考虑重新引入试衣或效果图：

- 用户已形成稳定留存
- 推荐文本本身已被证明可信
- 单套试穿确认已证明能提升决策完成率
- 图像能力不会拖慢核心体验

## 十二、最终判断

这个项目的首版不该再试图证明“AI 能生成多少图”，而应该先证明：

**当用户只录入少量常穿衣服时，系统能不能真的帮她少想一点，先给出 1 套可信方案，再在需要时用本人照片完成最后确认。**

如果这一点成立，后续再谈更强的视觉效果、更深的个性化和更复杂的 AI 能力才有意义。
